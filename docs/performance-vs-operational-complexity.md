# 성능 이득 vs 운영 복잡도 의사결정 기준

## 기본 원칙

**성능 이득이 있어도 운영 복잡도가 구조적으로 커지면, 기본값은 보수적으로 컷합니다.**

프론트엔드 도구 체계에서 더 중요한 것은 한 번 빠른 결과보다:

- 계속 굴릴 수 있는 구조
- 새 사람이 들어와도 이해 가능한 구조
- 장애가 나도 복구 가능한 구조
- 문제 원인을 설명할 수 있는 구조

입니다.

즉,

> **조금 더 빠름보다 계속 관리 가능함이 보통 더 중요합니다.**

---

## 판단 기준 3개

### 1. 성능 이득이 얼마나 큰가

작은 평균 개선보다 **노이즈를 넘는 분명한 개선**인지가 더 중요합니다.

- **5~10% 개선**: 의미는 있지만 애매한 구간. 복잡도 증가가 조금만 있어도 보수적으로 봅니다.
- **10~20% 개선**: 후보로 올릴 수는 있지만, 재현성과 운영 단순성이 같이 필요합니다.
- **20~30%+ 개선**: 구조 변경을 진지하게 정당화할 수 있는 구간입니다.
- **압도적 개선**: 복잡도 증가를 일부 감수할 수 있는 구간이지만, fallback/가시성은 여전히 필요합니다.

핵심은 다음입니다.

> **작은 평균 이득보다, 반복 실행에서 안정적으로 재현되는 이득인지가 먼저입니다.**

### 2. 복잡도가 어디에 쌓이는가

복잡도의 총량보다 **번지는 범위**가 더 중요합니다.

상대적으로 관리 가능한 복잡도:
- 설정 파일 1개 추가
- 특정 명령에만 국한된 wrapper/helper
- 문서/테스트 범위가 제한적인 변경

위험한 복잡도:
- runner / pipeline / CI / hook / fallback / monitoring 전반으로 번지는 변경
- 플랫폼별 예외 처리 증가
- 상태 추적/복구 로직 증가
- 특정 사람만 이해하는 암묵지

핵심은 다음입니다.

> **복잡도가 국소적이면 관리 가능하지만, 시스템 전체로 번지면 장기적으로 거의 항상 비싸집니다.**

### 3. 실패했을 때 되돌리기 쉬운가

다음 질문에 답할 수 있어야 합니다.

- 쉽게 끌 수 있는가
- 자동 fallback이 있는가
- rollback이 쉬운가
- 결과 해석이 명확한가
- 장애 원인 추적이 되는가

핵심은 다음입니다.

> **빨라져도, 실패했을 때 설명도 복구도 안 되면 채택 기준은 훨씬 더 엄격해야 합니다.**

---

## 실무용 채택 기준

### 채택해도 되는 경우

- 성능 이득이 **크고 명확함**
- 운영 복잡도 증가는 **국소적임**
- 문서화 / 검증 / rollback 가능
- 특정 개인 의존도가 낮음
- fallback / 상태 가시성이 있음

### 채택하지 않는 것이 좋은 경우

- 성능 이득은 있는데 **설명하기 어려움**
- 운영 체계가 더 복잡해짐
- 장애 시 디버깅이 어려움
- “아는 사람만 아는 시스템”이 됨
- 평균은 좋아 보여도 분산이 큼

이런 경우는 장기적으로 거의 항상 손해입니다.

---

## 현재 fooks 상황에 대입하면

최근 benchmark/probe 기준으로 보면:

- one-shot CLI baseline은 이미 건강한 편입니다.
- direct helper는 매우 빠르지만, 실제 제품 경로(launcher → helper)는 아직 일관된 승리를 증명하지 못했습니다.
- thinner launcher/helper 방향은 유망한 연구 후보이지만, 아직 기본값으로 승격할 단계는 아닙니다.

따라서 현재 기본 입장은 다음과 같습니다.

> **production baseline은 one-shot CLI로 유지하고, thinner launcher/helper는 실환경에서 재현 가능한 성능 이득 + 자동 fallback + 디버그 가시성을 함께 증명할 때만 다시 execution 후보로 올립니다.**

---

## 현재 기본 운영 원칙

- **작은 성능 이득 < 계속 관리 가능한 구조**
- **유망한 기술 데모 < 실제 제품 경로에서의 재현 가능한 승리**
- **속도 개선 < 운영 가능한 속도 개선**

즉 지금은 helper를 즉시 채택하는 단계가 아니라,

> **one-shot CLI를 안정적인 기본선으로 유지하면서, thinner launcher/helper를 엄격한 reopen 기준 아래 두는 단계**

로 보는 것이 맞습니다.
