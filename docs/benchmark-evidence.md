# Benchmark evidence and claim boundaries

This page is the public, curated benchmark evidence summary for `fooks`.
It intentionally replaces links to raw generated report artifacts in the public
README and npm package. Raw benchmark reports, patches, diffstats, and runner
outputs are generated evidence and should stay local or be regenerated by the
benchmark harness.

## Current public claims

`fooks` may claim:

- Codex-focused repeated-file hook support through `fooks setup`.
- Prepared-context compression mechanics for supported `.tsx` / `.jsx` files.
- Promising file-targeting behavior in targeted frontend benchmark runs.
- Conservative benchmark/proxy evidence, with clear boundaries.

`fooks` must not claim yet:

- Billing-grade provider token savings.
- Stable direct runtime-token or wall-clock wins across task classes.
- Automatic Claude or opencode runtime-token savings.
- Universal file-read interception outside the documented Codex hook path.

## Latest summarized evidence

### Initial Codex-oriented proxy snapshot, 2026-04-14

A five-task frontend benchmark sample showed a prepared-context proxy estimate
reduction from roughly 2.1M estimated context tokens to roughly 450K estimated
context tokens, with 5/5 successful runs in both variants. This is useful as a
prepared-context compression signal, not as billing telemetry.

### Direct-Codex Formbricks follow-up, 2026-04-16

A later direct-Codex Formbricks follow-up found that runtime-token and time wins
were not stable. Across six paired runs, `fooks` used more runtime tokens in
three pairs, and median runtime-token reduction was negative. Treat this as a
regression signal until future benchmark classes repeatedly demonstrate stable
wins.

### Layer 2 R4 applied-code diagnostic, 2026-04-22

A pre-launch repeated R4 applied-code diagnostic attempted seven matched
vanilla/fooks pairs after the proposal-only smokes, validator hardening, and
applied acceptance gate. Only four pairs passed acceptance in both modes, so the summary classification is
`insufficient-accepted-pairs`. Within the accepted pairs, fooks supplied a much
smaller prompt (median 88.2% reduction), but CLI-reported runtime tokens and
latency regressed (median runtime-token reduction -25.5%; median latency
reduction -14.4%). The CLI runtime-token field is not provider billing
telemetry. This result blocks provider billing-token, stable runtime-token, and
stable time-win claims and motivates another repeated run after prompt/validator
hardening.

A follow-up risk-closure rerun on the same date was stopped after three matched
attempts because zero pairs passed acceptance in both modes. Its summary also
classified the run as `insufficient-accepted-pairs`; runtime-token and latency
medians were intentionally absent because there were no comparable accepted
pairs. Treat this as additional negative diagnostic evidence, not as a win or a
replacement for a later multi-task applied-code benchmark.

## Reproducing or extending evidence

The benchmark harness source remains in `benchmarks/`. Generated outputs under
report/result directories are intentionally ignored and should not be committed
unless converted into a curated public summary like this page.

When updating this page, keep claims aligned with `docs/release.md` and avoid
linking public docs to raw generated artifacts.
